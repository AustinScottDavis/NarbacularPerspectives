CS 184: Computer Graphics and Imaging

Narbacular Perspective(s) [Proposal]

Paxtan Laker, Austin Davis, Emily Nguyen






Overview

Imagine Valve’s Portal, but users are allowed to create portals of different sizes? What do you think would happen when you stepped through the larger portal and exited the smaller portal?

We are striving to make a VR demo that answers this question by creating a system in Unity to allow users to create portals of varying sizes that will scale the size of objects passed through them (relative to the ratios of their area). We imagine that this system can create unique puzzle-like scenarios where users have to use portals for teleportation and perspective shifts simultaneously. We also find the implementation implications incredibly intriguing from a computer graphics perspective and want to put our Unity / CS 184 skills to the test.

Problem Description(s)

We are striving to put a modern twist on the classic concepts found in Valve’s Portal. Both from a graphics implementation and conceptual perspective, the idea of portals is incredibly intriguing. The main focus of this project will be to create a demo where the user can place multiple portals within the world through which they can place objects, or walk through themselves. However, we will be going a couple of steps further and allowing for the use of portals of different sizes, as well as making the demo playable in first person in VR. The idea behind having portals of different sizes is that the scale of whatever passes through the portal will be adjusted to match the difference in size from the original portal to the new one.

We think this problem is important because it is really cool. Seriously though, this idea just sounds fun and intriguing. That's why we are doing it.

One difficult part of this project will be figuring out how to accurately and realistically map one side of the portal to the other. Another challenge will be to solve recursive portals. Honestly, this whole project is incredibly challenging because the implementation raises A LOT of questions. It would be really arrogant to claim that we already know what the most challenging part is before starting. There are SO MANY conceptual problems you can think of fairly immediately from a game engine perspective. (We've listed some below....)

  • When an object is passing through the portal (halfway through) do we need two mesh representations spawned in the game worlds on both sides? Or is there some kind of clever way we can do shading and rendering tricks to translate half to one side of the portal and half to the other?

  • When looking through portals that obscure one another, should we render infinitely through the portals (imagine a webcam pointed at the result of the webcam) ?

  • How do you implement ray casting through a portal? Unity has built in function for ray casting we will probably have to rewrite ourselves for this

  • Our stretch goal is to allow the user to turn any surface into a portal → I’m not sure where to even start with this one!

  • Can a user peer through or go through a portal of any size? We would have to figure out how to scale the user as they go through as well.

Our MVP (Plan)

  • Players should be able to interact in a virtual reality environment, create portals on command, and see and move through portals

  • Working Portals in Unity that support teleportation and size-scaling for AT LEAST the player themself (specifics listed below)

    • Smooth transitions from one portal to the other

    • The ability to see through portals

    • The ability to raycast through the portals

    • The ability to walk through the poratls

    • The ability to create portals of any size (within our minimum - maximum constraint) anywhere

    • Some way of handling recursive portal rendering when they are obscuring each other

    • Figuring out how we want to handle the "backside" of a portal


  • Full VR integration along with some kind of locomotion that pairs well with the portals

    • The ability to deal with moving through portals that “don’t fit” the player

    • The ability to potentially raycast through the portals to teleport (boring)

    • Supports all relevant headsets


Why we think we can accomplish these goals

  • We believe this class has taught us enough of rendering to dive deeper into the Unity nitty gritty than we could before

  • This project involves a lot of consideration of camera space/world space positions and translations like those we have already been doing in this class.

  • All three of us are facilitators for the VR Decal, so we don’t think the VR component of this project will be all that hard to do. While cool, VR is not the primary focus of the project because of this.

What we hope to deliver (stretch goals)

  • The ability to transform any surface into a portal, rather than creating a portal in world space from scratch

  • A unique solution to the backside of the portal (this is kind of an unanswered questions)

  • Cool puzzle implementations to pair with our unique portal system

How are we measuring?

There isn't really anything to measure. Do portals work? Does it run well enough to match most VR experiences? Is everything smooth? If yes to all of these ---> 100%
